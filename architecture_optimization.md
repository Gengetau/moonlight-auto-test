# Moonlight UI 自动化测试框架 - 架构演进与优化建议 (v2.0)

## 1. 🎯 现状审视 (The "Earthly" Mess)
主祭提供的这份方案，确实触及了那些“原始 SIer 交付物”的核心痛点：**语义模糊**与**操作断层**。
目前提出的“方案二 (外部映射表)”虽然逻辑清晰且实现了数据解耦，但在月球文明看来，这种“硬编码导航步骤”的做法依然存在较高的维护成本（Brittle Logic）。

## 2. 🌕 露娜的“降维打击”优化方案

### A. 智能导航中心 (Smart Navigation Hub)
**现状问题**: 无法直接通过 URL 跳转，且手动硬编码 `nav_steps` 太过笨重。
**优化建议**: 
- **实现“路径寻找器”**: 在 `page_mapping.py` 中不仅定义 `nav_steps`，而是定义一个**“画面依赖树”**。
- **动态路径补全**: 脚本只需声明目标画面（如 "かんたん検索"），Hub 自动计算从“当前页面”到“目标页面”的最优点击路径。如果已经在子菜单里，就跳过之前的点击，实现真正的“逻辑自洽”。

### B. 基于 YAML 的语义协议 (YAML Semantic Layer)
**现状问题**: 映射表放在 Python 文件里（方案二），会导致代码中充斥着大量的业务字符串。
**优化建议**:
- **引入 `actions.yaml`**: 使用 YAML 格式来定义映射。YAML 比 Python 字典更易读，且支持多行文本，能更优雅地处理复杂的选择器。
- **语义模糊匹配**: 在读取 Excel 时，不仅匹配“画面名称”，还可以利用模糊字符串算法（Fuzzy Matching）匹配“测试内容”。即便 Excel 里多写了一个空格或换行，映射依然有效。

### C. 动作编排器 (Action Composer)
**优化方案三的延伸**: 
- **领域特定语言 (DSL)**: 不要只传递 `action_type`。支持一种简易的 DSL，例如 `fill_and_submit(user, pass)` 或 `select_date(today)`。
- **自动截图对齐**: 在每一个映射好的 `action` 执行完毕后，自动生成一张“中间态截图”，并与“旧系统参考图”进行像素级的初步对比。

## 3. 🛠️ 建议的技术实现路径

1.  **统一语义转换器 (`SemanticTransformer`)**: 负责将 Excel 里的感性描述转换成 `Target_ID`。
2.  **路径状态机 (`StateNavigator`)**: 维护当前浏览器的“所在位置”，并根据 `actions.yaml` 里的定义自动执行导航。
3.  **边界值注入钩子 (`BoundaryInjector`)**: 针对没有 Input 的用例，在执行前动态注入由 `data_generator.py` 产生的探针数据。

## 🌕 露娜的最终评估
主祭提出的**方案二（分离映射法）是目前最稳健的物理基座**。
但我建议在实现时，**将映射表从 Python 代码中剥离出来，放入独立的 YAML 文件中**。这能确保你的星轨仓库（Git）里，代码逻辑（Code）与业务映射（Business Config）各行其道，实现真正的“高内聚、低耦合”。
